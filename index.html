Milestone 5

Date: 2024.05.26

Primary Focus

The primary objective of Milestone 4 is to implement the functions that belongs to protocol layer. In current stage, we just do not need so many functions in protocol layer to maintain our VBB mechanism work (we put lots of the services provided in initial web services in Java). So, we just implemented the functions related to verification in Python.

In fact, there are multiple parts for verification in blockchain. They are all based on the characteristics of hash function.

A hash algorithm is an algorithm that converts an input message of arbitrary length into a fixed-length output message. Hash values have the following characteristics:

  • Uniqueness: The hash value generated by the hash algorithm is unique for different input data.
  • Fixed length: The length of the hash value generated by the hash algorithm is fixed.
  • Irreversibility: The content of original input data cannot be deduced from the hash value.
  • Collision resistance: It is difficult to find two identical hash values in the input space.
  • Used for verification: that is, the data we use to vote: the input data is the same, the output hash value is the same; the input data is different, the output hash value is also different.

Verification between blocks part

In the blockchain, each block contains a block header (Block Header), which contains metadata about the current block and a hash value pointing to the previous block. The hash value of the previous block is actually a “pointer” to the previous block, which is used to establish links between blocks.

We can verify whether the chain has been tampered with by verifying whether the hash value stored in the header of this block matches the hash value of the previous block. The existence of this pointer ensures the continuity and integrity of the blockchain, because each block contains the hash value of the previous block, so no one can tamper with the data in the blockchain, otherwise it will be destroyed Blockchain structure.

Verification between blocks parts

When we consider the integrity verification of the information inside the block, we have another method. To verify whether the block content voting information has been tampered with, we use Merkle Proof, which is a mechanism used to verify that any votes in the block actually exists and has not been tampered with.

This is based on the structure of Merkle Tree – a binary tree in which each leaf node is the hash value of a vote, and each non-leaf node is the result of merging the hash values of its child nodes and then hashing it. Its construction process is as follows：

  1. Get the vote hashes: First, we need to know all the hashes you want to verify.
  2. Get the Merkle Path: Starting from vote hashes, find a path to the root of the Merkle tree. There will be a series of hashes along this path that are used to calculate from the leaf nodes to the root node.
  3. And then Recalculate and compare the root hash:
  4. Verify root hash: If the recalculated root hash matches the Merkle root stored in the block header, then we believe the voting data was not tampered with.

This means that by verifying the hash value of the Markov tree, any illegal block operations can be effectively detected.

Milestone 2

Date: 2024.03.25

Primary Focus

The primary focus of Milestone 2 was to develop the basic frontend and backend code of the voting platform, marking a crucial step forward in the realization of our project.

Frontend Development

Designed the basic framework and functions of the front-end web page, the system is roughly divided into two major functions: poll management is used to initiate new polls and view the results of previous polls; user management is used to give higher privileges to the account management of the site’s users, and in the future, it will be added to the deletion and upgrading of other functions.

Current Web Development

At present, we have built the login and registration interface and the home page:

login and registration interface as shown in the figure, users can choose to log in directly to the existing account or register a new account; successfully logged into the system, the home page of the site will display the user’s basic information and permissions, the header part of the home page has our main functions, users can click on the jump, as well as in the home page can be directly clicked on the initiation of a new vote, it’s more convenient! The homepage header part has our main functions, users can click to jump, and in the homepage can directly click to launch a new poll, more convenient.

Backend Development

Technology Choice

  • Framework: Spring Boot was chosen for its ability to quickly build the project and focus on the implementation of business logic.
  • Persistence Layer: MyBatis-Plus provides convenient methods that greatly simplify the development of the data access layer.
  • Database: MySQL is used for efficient data storage and management.

Implementing Functionality

The project structure contains:

  • Controller Layer: Interface logic
  • Manager Layer: Business logic
  • Service Layer: Data logic

At this stage, important functions such as user management, login and registration, and voting management were successfully implemented. The design ensures high cohesion and low coupling of the backend code, improving system scalability.

Deployment and Operation

The project was deployed on Huawei Cloud servers. The deployment process involved:

  • Configuring Java, MySQL, and other runtime environments.
  • Uploading the project as a JAR file via Maven commands.
  • Opening relevant ports to ensure normal access and operation.

milestone 1

2024.3.16

We designed the whole architecture of our VBB mechanism from high level - three layers architecture to low level - account functions. Three are two key points.

  1. Centralization: There’s no need for distributed voting, as it doesn’t match the actual situation, is inefficient, and nobody wants to install software. Direct access to the official website via the web is preferred.
  2. Bind accounts to voting events, rather than to individuals: This is because binding accounts to individuals may lead to the loss of private keys over time, especially if one participates in voting events repeatedly over the years.

And we have primarily allocated two types of accounts, T1 account (voter account )and T2 account (manager account).

T1 Level accounts, official accounts, have the right to create voting events and issue T2 level accounts.

T2 Level accounts, temporary accounts, valid only for the duration from the start to the end of a specific voting event.
